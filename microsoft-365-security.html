1. MFA Fatigue Attack (Approve Spam)

TOP IN 2026

Scenario: Hacker gets your password from breach (you reuse @mydomain + old password). At 2:13am they trigger login, you get 20x “Approve sign-in?” push at night. Tired admin at client in Menlyn tapped Approve at 06:12 to make it stop.

Damage in SA: R87k stolen — attacker created mail rule to hide, sent new bank details to your customer. POPIA breach because they accessed payroll with ID numbers.

How NextGrid stops it:

  • Number Matching: user must type 2-digit code shown on login screen — no more blind Approve
  • Conditional Access: block logins outside SA, impossible travel (Pretoria → Lagos in 5 min = block)
  • Sign-in risk policy: high-risk token = force password reset + block
  • We disable SMS MFA entirely — SIM swap up in SA, Authenticator only

2. Session Hijacking / Token Theft (EvilProxy)

Scenario: Finance lady clicks “Microsoft: Your OneDrive is full” link. Site looks exactly like M365 login (even has correct logo). She logs in with MFA. Attacker steals session cookie — no password needed next time. Even with MFA, they are in for 90 days.

Why MFA alone fails: Token = already authenticated. Happening via WhatsApp now too: “Your parcel invoice.pdf”.

How we stop it:

  • Conditional Access + Entra ID Protection: token replay from new device/IP = challenged
  • Require compliant / hybrid-joined device for Outlook web, enforce sign-in frequency 4h
  • Safe Links rewrites link so it never reaches real phishing site — scanned at click-time globally
  • Continuous Access Evaluation enabled — revokes session within 15 min on risk signal

3. Business Email Compromise (CEO Spoof)

R6.2M avg loss SA

Scenario: Email from johan@nextgridtechnologies.co.za but actually johan@nextgridtechnologieS.co.za (capital S/i trick) to accounts: “Please pay supplier urgently – EFT details changed”. Uses real thread history lifted from compromised inbox. Accountant pays R145k.

Local twist: During loadshedding, staff rush — less checks. Attackers monitor Inbox Rules: they hide their sent mail.

How we stop it:

  • Anti-impersonation policy: protect your CEO, finance, HR display names + domains
  • SPF hard fail + DKIM 2048 + DMARC reject → spoofed mail never delivered, anywhere
  • External tag: bright banner “EXTERNAL – do not trust bank details”
  • Mailbox alert: alert on Inbox Rule creation, forwarding to gmail = auto-block + alert to 076 948 9154 monitored inbox

Email Authentication: SPF / DKIM / DMARC in plain English

SPF
Return-to-sender list. “Only these servers can send as @yourdomain”. We set -all hard fail. If spammer tries to send as you from Gmail server, receiving bank’s Office365 drops it. Without it, your invoices go to spam and clients get spoofed.
DKIM
Digital signature, 2048-bit. Proves mail wasn’t changed in transit. Like wax seal. We rotate keys every 6 months. Needed for trust + deliverability to Gmail/Yahoo new 2024+ rules.
DMARC
Bouncer instruction. “If SPF or DKIM fails, reject”. We start p=none + reports, 48h later p=reject. We read aggregate reports daily — see who tries to spoof you (often Botswana, SA hosting).
Common SA mistake: IT guy sets SPF ~all softfail + no DMARC. Result: you CAN still be spoofed, and POPIA “safeguards” weak. We fix DNS at registrar (Afrihost, xneelo, etc) same day.
